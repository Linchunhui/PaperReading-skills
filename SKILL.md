# PaperReading: 读论文

读论文不是做学术，是猎取思想。把别人的发现拆解成自己能用的认知。

## 总目标（先看这个）

输出一份*可直接当笔记长期保存*的结构化解读 HTML，让一个**不懂这个领域的聪明人**读完能复述：

1. 论文在解决什么问题（具体到一个例子）
2. 作者用什么招数解的（机制 + 设计选择的理由）
3. 核心发现是什么（包括最反直觉的副发现，往往是论文里最 fascinating 的部分）
4. 你（读者）能带走什么洞见 + 能动手的启发

如果输出在任何一点上让外行卡住，就是失败。**凝练只在 HTML 大标题上追求；正文该展开就展开**——目标不是短，是让人从不懂走到懂。

最终交付一个 HTML 文件（自包含单文件、可独立打开、内嵌配图与 SVG 可视化、KaTeX 渲染公式）。保存路径默认与原 PDF 同级目录；若来源是 arxiv URL 等无本地 PDF 的场景，先把 PDF 下载到 `~/Documents/notes/pdf/` 再把 HTML 放到同目录。

## 格式约束

### ASCII Art（论文里的简单结构图）

正文中的简易示意图（流程、结构、对比）可以用纯 ASCII 字符。允许：`+ - | / \ > < v ^ * = ~ . : # [ ] ( ) _ , ; ! ' "` 和空格。禁止 Unicode 绘图符号。

更正式的、需要承重表达的图（架构图、数据流、消融对比）用 *SVG*——见「工具箱 → SVG 可视化」。

### 文件命名

读论文产物：
- *配图（论文截图）*：`~/Documents/notes/images/{identifier}--paper-{简短标题}-fig{N}.png`
- *PDF（无本地 PDF 时下载）*：`~/Documents/notes/pdf/{identifier}--paper-{简短标题}.pdf`
- *HTML 输出*：
  1. 若有本地 PDF → 与 PDF 同级目录，文件名 `{简短标题}.html`
  2. 若需下载 PDF → `~/Documents/notes/pdf/{identifier}--paper-{简短标题}.html`
  3. 纯网页来源（无 PDF）→ `~/Documents/notes/html/{identifier}--paper-{简短标题}.html`

工具：
- 时间戳：`date +%Y%m%dT%H%M%S` 用于 identifier
- 简短标题：方法名 / 核心概念名（如 `prerl`、`dsrl`、`lenvm`），全小写、英文缩写、用作文件检索

### HTML 大标题（h1）这一句怎么写

朴素直接：`{论文简短名} 阅读笔记` 即可。不追求凝练、不追求金句、不需要"灵魂句"——读者扫一眼就知道这是哪篇论文的笔记，就够了。

**模板**：

```
<h1>{论文简短名} 阅读笔记</h1>
<p class="subtitle">{论文原标题} — {第一作者等}, {年份}</p>
<p class="meta">{阅读日期} · <a href="{源链接}">原文</a> · <a href="{PDF 路径}">PDF</a></p>
```

**简短名怎么取**：

- 优先用论文里出现的*方法名 / 模型名*，保留原大小写：`SenseNova-U1`、`PreRL`、`LenVM`、`DeepSeek-R1`、`GPT-5`
- 如果论文没有方法名，用一个 2-4 词的英文核心概念：`Mixture of Experts Survey`、`Speculative Decoding`
- 如果是中文论文，可以直接用论文中文简称
- 不要在 h1 里加形容词、不要"惊人发现"、不要二次创作

**正例**：
- `SenseNova-U1 阅读笔记`
- `DeepSeek-R1 阅读笔记`
- `Speculative Decoding 阅读笔记`

**反例**：
- ✗「无中介，方一体」（过度凝练，看不出是哪篇）
- ✗「关于多模态统一架构的探索性研究阅读笔记」（学术腔且重复）
- ✗「SenseNova-U1：把 VE/VAE 全拆掉的统一模型 阅读笔记」（方法描述塞进标题）

**为什么朴素就够**：
- 标题的核心职责是*定位*——让读者半年后翻笔记能一眼找到自己想看的那篇
- 思想的凝练交给 Insight 章节去做，不要让 h1 同时承担"识别"和"金句"两个职责
- 朴素标题反而更好搜索、更好分享、更好整理

## 红线（每条必须过）

1. *口语检验* — 你会这样跟朋友介绍一篇论文吗？不会→改。学术腔是默认敌人
2. *零术语裸奔* — 术语首次出现必须落地（"value function = 给定当前状态预测未来累积奖励"）。如果必须用原文术语才能解释，说明还没懂
3. *短词优先* — 能用两个字说的不用四个字。「本文提出了一种新的框架」→「他们做了个东西」
4. *一句一事* — 每句只推一步
5. *具体* — 名词看得见，动词有力气。形容词能砍就砍
6. *开头给理由* — Motivation 第一句让人想知道答案
7. *不填充* — 删学术套话（「近年来随着...的发展」「值得注意的是」）。每句干活
8. *信任读者* — 说一遍够了。不重复结论
9. *诚实* — 论文有硬伤就说有硬伤。看不懂的部分说看不懂
10. *公式必翻译* — 每个 LaTeX 公式后必须用一句中文说明它在做什么。没翻译 = 没解释
11. *6 个月后的我看得懂吗？* — 自检：默想"半年后再打开这个 HTML，30 秒内能回想起核心吗？"——不能 → 重写
12. *外行优先于凝练* — 该展开就展开，不要为了"短"砍掉读者真正需要的铺垫。凝练的尺度只针对 h1；正文该多长就多长，节奏由让人懂的需要决定，不由"短"的洁癖决定

## 写作原则

四条核心原则，决定文章是"活人在说话"还是"机器在汇报"：

1. *一个锚点撑全文* — Motivation 里立的那个具象场景就是锚点。后面的方法详解、核心概念、洞见都在这个锚点上展开，每段都回到它，让读者一直待在同一个问题域里。换锚点 = 换地图 = 读者前面建立的直觉全丢。
2. *推理外显* — 模拟"一个人想明白的过程"，而非呈现"想明白之后的结果"。用"既然 A 是 B，那能不能 C 也是 D？"带读者一起推。让读者觉得结论差一步就是自己想到的
3. *变形替代定义* — 解释两个概念的关系时，把 A 连续变形成 B，不要说"A 和 B 是 XX 关系"。「把 LSTM 变形 → 看起来像 ResNet」比「LSTM 和 ResNet 是对偶的」有力十倍
4. *落点在能用* — 启发部分给出"这意味着你可以___"，而非"这让我们重新思考___"。读者读完要带走一个能动手的东西，不是一个值得沉思的感慨

## 工具箱（选用）

讲解论文时可以拿的工具，没有哪个是必须的：

- *类比* — 承重的，方法的关键组件都能映射上。沿着类比走一遍方法
- *ASCII 图* — 简易结构、流程、对比，正文里直接展示
- *SVG 可视化* — 见下方专节，承重的图都用 SVG
- *餐巾纸速写* — 「以前这么想，现在应该这么想」的并排对比（SVG 实现）
- *好问题* — 把论文解决的困境变成一个让外行也好奇的问题
- *递进例子* — 从简单到复杂，一步步搭建理解
- *反问入链* — 遇到隐含假设，用问题打开

### SVG 可视化（什么时候画、怎么画）

数字、对比、流程、关系——只要能用图说清楚比文字说得快，就画 SVG。SVG 内嵌在 HTML 里（无需外部资源），是这套笔记的*视觉骨架*之一。

*什么场景必画*：

1. *消融实验对比* — 「去掉 A 掉 5 点，去掉 B 掉 12 点」用横向 bar chart，比表格直观十倍
2. *主结果对比* — 在 N 个 benchmark 上 vs N 个 baseline，分组 bar / dot plot
3. *方法流程* — 数据如何流过模块，箭头 + 节点
4. *训练曲线* — Loss / accuracy 随 step 变化
5. *分布对比* — 「之前的分布是这样，现在变成这样」叠加分布图
6. *架构图* — 模型组件、连接关系
7. *因果链* — 「输入这个 → 触发那个 → 导致结果」的概念图

*数据准确性是 SVG 的红线*——画错数字比不画更糟：

- *每个数字必须有出处* — 画 SVG 前先在论文里把要用的数字定位到具体 Table N 或 Figure N，写一个临时记录（哪个数字来自哪张表的哪一行哪一列）
- *不要"凭印象"或四舍五入估算* — 论文写 88.14 就是 88.14，不能写 88、88.1、~88；论文写 0.940 就是 0.940，不能写 0.94
- *不要把"开源最佳"等同于"全部最佳"* — 仔细看论文表格分了 closed-source / open-source 两段，画图标注时必须区分
- *baseline 和对比对象要选论文里真实有的对比* — 不能把没在论文里直接对比过的模型硬拉进来对比；如果非要补充上下文，明确标注"非论文数据"
- *画完逐项核对一遍* — 把 SVG 里的每个数字和源表的对应单元格逐一对照，发现差异立刻修；这一步不可跳过
- *标题和说明也要准* — bar chart 的 "越长越好"/"越短越好"方向、坐标轴单位、对比组的含义，都要和论文一致

*视觉原则*：

- *一个图传达一个 insight* — 不要在一张图里塞五件事
- *简洁线条、最少颜色* — 主色一种、强调色一种、辅助灰，够了
- *标注必备* — 每条数据都要有数值标签，每个轴都要有标题，每个对比都要标"+5%"这种增量
- *响应式* — `viewBox` 而非固定宽高，能在手机上缩放
- *字体跟随* — 用 `font-family: inherit` 让 SVG 文字和 HTML 正文字体一致
- *亲手写而非工具生成* — 不要套 chartjs / d3 这种重库，直接写 SVG 元素，单文件可独立打开
- *末尾标数据来源* — SVG 里加一行小字 "Source: Table N / Figure N of the paper"，让读者能追溯

*简单 bar chart 模板（消融实验示意）*：

```svg
<svg viewBox="0 0 600 280" xmlns="http://www.w3.org/2000/svg" style="font-family: inherit;">
  <text x="20" y="30" font-size="16" font-weight="bold">消融实验：去掉某组件的影响</text>
  <!-- 横轴 baseline 线 -->
  <line x1="180" y1="80" x2="180" y2="240" stroke="#999"/>
  <!-- 各项 bar，每条配上数值 -->
  <rect x="180" y="90" width="320" height="24" fill="#4a6fa5"/>
  <text x="510" y="107" font-size="13">+完整方法 (78.5)</text>
  <rect x="180" y="125" width="280" height="24" fill="#4a6fa5" opacity="0.7"/>
  <text x="470" y="142" font-size="13">−组件 A (74.2)</text>
  <rect x="180" y="160" width="200" height="24" fill="#c97064"/>
  <text x="390" y="177" font-size="13">−组件 B (66.1)</text>
  <text x="20" y="270" font-size="11" fill="#666">越长越好；组件 B 是关键</text>
</svg>
```

哪些数据值得画：实验表格里抽出读完后让人"哇"的那条对比；方法描述里如果有"先 A 再 B 再 C"的步骤；消融对比里增量明显的那几条。其他凑数表格不必画。

## 执行

### 1. 获取内容

- arxiv URL → WebFetch 拿到 abstract 和正文；同时记下 PDF URL（通常是 `arxiv.org/pdf/xxx.pdf`）
- 本地 PDF → Read（注意 pages 参数限制）
- 论文名称 → WebSearch 找到 arxiv / 官网链接，再走 URL 流程

确保拿到：标题、作者、摘要、Related Work（前作的引用上下文）、Method（章节结构与公式）、Experiments（主结果表 + 消融表）、Limitation（如果有）。

*配图提取*：识别承载理解的关键图——架构图、流程图、关键示意图、失败案例可视化、消融对比图。提取并保存：

- arxiv → 优先访问 HTML 版（`arxiv.org/html/{paper_id}`）找原始图片 URL，WebFetch / curl 下载（这种方式得到的是高分辨率原图，边界天然完整）
- PDF → 截取含图页面保存为图片
- 路径：`~/Documents/notes/images/{identifier}--paper-{简短标题}-fig{N}.png`

*结构图 / 架构图截图必须边界完整、清晰可读、文件大小适中*——三条同时满足才行：

**(a) 边界完整**：不能让读者看一张被裁掉一角的图来理解方法

- *截图前先定位完整边界* — 找到图的标题（Figure N: ...）和图正下方的 caption，框选范围必须包含从图正上方一点点空白到 caption 末尾，不能切到任何元素
- *优先用 HTML 版图源而非 PDF 截图* — arxiv HTML 版的图通常是独立 PNG/SVG，下载即完整；PDF 截图容易因为页面跨页、旁边有公式、缩放比例不对而被裁
- *如果只能 PDF 截图*：用足够大的截图区域（宁可多留白，不要切边）；如果图横跨两页，分别截取后竖向拼接；截完*打开图片肉眼检查*，确认四周没有截到一半的字、没有遮挡的图例、没有缺失的箭头终点
- *复杂架构图建议两张*：一张全景（整体结构）+ 一张关键子模块特写。读者先看全景建立心智模型，再看特写理解细节

**(b) 清晰可读**：每个字、每个箭头都能看清

- *截图 DPI 至少 200*：用 `pdftoppm -png -r 200 ...` 而不是默认的 72 DPI；模糊的图等于没有
- *最终宽度 1200-1600 px*：太窄文字糊、太宽页面卡顿
- *肉眼检查*：在浏览器里 100% 缩放看文字是否清晰；遇到 PDF 内嵌的小字图，单独再截一张放大版

**(c) 文件大小适中**：HTML 不能因为一张图就肥到打不开

- *单张图压缩后 ≤ 300 KB*；总图量（base64 嵌入后）控制在 ≤ 2 MB
- *转 JPEG 而非保留 PNG*：除非是线条图截图（PNG 锯齿更少），照片类一律 JPEG quality 80-85 优化
- *压缩流程*（推荐用 PIL 一段脚本搞定）：

```python
from PIL import Image
img = Image.open(src_png).convert("RGB")
# 限制最大宽度避免过大
if img.width > 1400:
    ratio = 1400 / img.width
    img = img.resize((1400, int(img.height * ratio)), Image.LANCZOS)
img.save(out_jpg, "JPEG", quality=85, optimize=True)
```

- *base64 内嵌前再核对一遍*：`ls -la` 看每张图的 KB 数；超出预算就降 quality 到 75 或降宽度到 1200

不是所有图都值得保存——装饰性图、无信息量的曲线图通常可以跳过。挑承重的（一般 3-6 张就够），多了 HTML 会臃肿、读者也疲劳。但凡你决定保留的图，都必须三条同时过：边界完整、清晰可读、大小适中。

*PDF 下载*（仅当来源是 arxiv URL 等无本地 PDF 时）：

```bash
mkdir -p ~/Documents/notes/pdf
curl -L -o ~/Documents/notes/pdf/{identifier}--paper-{简短标题}.pdf {pdf_url}
```

### 2. 完整解读：结构化笔记

笔记的*完整解剖学*——产出一份让外行能跟、半年后还能查的结构化解读。每节用大白话，避免学术腔；该长就长，不要为了"短"砍掉读者真正需要的铺垫。

#### 摘要（这篇工作在讲什么）

200-300 字的整体介绍，让读者*在进入 Motivation 之前先有一张全景地图*。这一节不是论文 abstract 的翻译，是你读完之后用大白话给一个聪明外行做的导览。

包含四个要点（不一定按顺序，融在自然行文里）：

1. *做了什么* — 这篇论文提出 / 研究的是什么（用一句话定位：「这是一篇关于 XX 的工作」）
2. *解决什么问题* — 一句话点出针对的核心痛点（细节留给 Motivation 展开）
3. *核心方法* — 用一句话概括方法机制（细节留给方法详解展开）
4. *关键结果* — 主要发现 / 提升 / 反直觉结论（细节留给实验结果展开）

写作要求：
- *不剧透洞见* — 摘要给"这篇做了什么"，洞见是"读完我看到了什么"，分开
- *不堆术语* — 必须出现的术语首次落地解释；外行扫一遍能跟上
- *200-300 字* — 短了讲不清全貌，长了就成了 Motivation 的预演。卡在这个区间
- *自带钩子* — 末尾留个让人想往下读的引子（「但作者发现……」「有意思的是……」）

写完后回头检查：单读摘要这一段，能不能让人对论文产生*三句话级别*的认知？「哦，这是做 XX 的，他们发现 YY，效果是 ZZ。」能 → 通过；不能 → 重写。

#### 一、Motivation（这工作为什么值得做）

三段大白话，每段一个具体场景，不要写"为提升 XX 性能"这种空话：

1. *现实痛点* — 当前要解的是什么真实问题？给一个具象场景，让读者"看见"痛点。例：「你让模型写一篇 100 字的产品介绍，它写了 800 字还在收不住——明明任务描述写了字数」
2. *前人卡在哪* — 之前的工作做到了什么、卡在哪？卡点的*本质*是什么——是数字差，还是思路上的盲区？
3. *作者的入口* — 本论文从哪个角度切进来？为什么这个角度有戏？

这一节立的具象场景就是后面*所有节的锚点*。换锚点 = 读者前面建立的直觉全丢。

#### 二、现存问题（前作的具体短板）

列 3-5 个前作的具体问题，每个一句话「问题是什么 → 后果是什么」：

- 问题 A：xxx → 导致 yyy
- 问题 B：xxx → 导致 yyy

不要泛泛说"已有方法效果不好"——必须落到具体机制上的具体短板。这一节是 2.1 第二段的展开。

#### 三、相关工作翻译

挑论文 Related Work 里最重要的 3-5 个先驱方法，每个 2-3 句翻译解释：
- 谁做的、做了什么
- 解决了什么、留下了什么
- 和本论文的关系（继承？反对？补充？）

用大白话重写，不照搬论文措辞。读完应能建立"这方向上前人都试过什么"的地图，免去读者去翻原始论文的成本。

#### 四、方法详解

按论文章节结构展开（一般是 Section 3 / Method）。每个子模块讲清三件事：

1. *做什么* — 模块的输入输出
2. *怎么做* — 核心机制；必要时用 LaTeX 公式辅助：
   - 行内：`$L = \mathbb{E}[\log p(y|x)]$`
   - 块级：`$$L_{\text{total}} = \sum_i w_i L_i$$`
   - 写完公式后*必须用一句中文翻译*这个公式在做什么——不解释公式 = 没解释方法
   - *公式必须能完整渲染*——这是硬要求。书写时遵守以下规范，否则 KaTeX 会渲染失败或显示错乱：
     - *转义符号正确* — 下划线 `_` 在数学环境内是下标符号，不要在公式里写未转义的 markdown 强调符；公式外的下划线如果会被 markdown 误解析，用反斜杠转义 `\_`
     - *配对完整* — 每个 `$` 必须有配对的 `$`，每个 `$$` 必须有配对的 `$$`；`\left(` 必须配 `\right)`；`\begin{aligned}` 必须配 `\end{aligned}`
     - *不混用未支持的 LaTeX 包* — KaTeX 支持的命令是 LaTeX 的子集，不要用 `\usepackage`、`\newcommand` 自定义、不要用 `\mathds`、不要用未声明的环境（如 `\begin{cases}` 可以但 `\begin{equation}` 不行）。常见替代见 https://katex.org/docs/supported.html
     - *中文不要混进数学环境* — 公式块里全英文/符号；解释中文写在公式*外面*。`$L_{\text{loss}}$` 这种用 `\text{}` 包裹的英文是允许的，但中文字符塞进 `\text{}` 里在某些 KaTeX 版本上会出错
     - *复杂多行公式用 `aligned`* — 不要用 `eqnarray`（不支持），用 `$$\begin{aligned} a &= b \\\\ c &= d \end{aligned}$$`，注意 markdown 转义里换行需要 `\\\\`（四个反斜杠）才能渲染成 LaTeX 的 `\\`
     - *写完自检* — 生成 HTML 后*必须打开浏览器肉眼确认每一条公式都正常渲染*，看到 `$...$` 原文裸露、看到红色报错框、看到符号缺失，立刻回去修。一条没渲染好就是整段没渲染好
3. *为什么这么做* — 设计选择背后的考量。这一条最容易被忽略，但最值钱

*配图嵌入*：从 2.1 步骤里提取的图，按论文结构嵌入对应位置（用 `<img src="../images/...">` 相对路径，或转 base64 内嵌）。不要堆图，每张图都要在正文里被讲到（"如图 1 所示——你看 X 模块怎么处理 Y..."）。

*SVG 流程图*（推荐）：方法整体的数据流、模块连接，用 SVG 画一张总览图放在本节开头。让读者先看到全貌再看细节。

每个子模块用类比和大白话，回到 2.1 的锚点场景上讲：「在我们刚才那个场景上，这个模块在做什么？」

#### 五、实验结果

*主结果一句话* — 在哪些 benchmark 上、达到什么水平、相对 baseline 提升多少。

*关键消融一两句话* — 重点是提炼"什么动作带来什么收益"的因果链：
- 哪个组件最关键？（去掉它掉多少点）
- 什么配置下效果最佳？
- 什么情况下方法失效？

*SVG 可视化*：把表格里最有冲击的对比抽成 SVG bar chart——见「工具箱 → SVG 可视化」。一张消融对比 SVG 比把整个 LaTeX 表格搬过来有用十倍。

目标是给读者*启发性结论*——读完应该能脱口说出"哦原来 X 这么做能涨这么多"。

#### 六、总结

三句话以内：
1. 这篇论文最核心的贡献
2. 它打开了什么新可能
3. 它留下了什么未解的问题（如果作者自己提了 limitation，转述；没提，自己看）

#### 七、Insight（按需出现，不强求）

整篇论文最值钱的往往就一个点——作者真正找到的那颗新结晶。如果这篇论文确实有这样的结晶，用一句话把它说出来；如果没有，这节可以略掉，或者直说「这篇论文是工程改进，没有认知层面的新发现」。

*不要硬挤*。读完一篇没有思想火花的工程论文，硬写一段强行升华的 Insight，比不写更糟——它会污染读者对"什么才算真正洞见"的判断。

如果写：

- 用一句话说出来。这句话应该让读者觉得「这个想法我可以带走」，而不是「哦，论文说了这么个事」
- 检验标准：把这句话单独抽出来，脱离论文上下文，它还有没有力量？如果只是在复述论文结论，那不是洞见。洞见是你读完之后自己看到的那个东西——论文里未必直说，但逻辑指向它
- 数量上一般 1 条就够；偶尔有论文真的同时打开了两件事，可以写 2 条；超过 3 条几乎一定是在掺水

如果写了 Insight，再回头炼 HTML 顶部的 h1——h1 是这段话取骨而成。如果没有 Insight，h1 就退到论文核心方法或核心发现的凝练，按 h1 写作规则照常处理。

#### 八、启发（按需出现，不强求）

如果这篇论文确实让你对自己手头的事产生了能动手的连接，就写；如果只是「读了挺有意思但用不上」，这节可以略掉。落点必须在"能用"，不在"能想"——给出"这意味着你可以___"，而非"这让我们重新思考___"。

用三个视角试探连接，命中几个写几个，全没命中就略过整节（或一句话说明「暂无可直接迁移的启发」即可）：

- *迁移* — 论文的某个机制 / 视角能移植升级你体系的某个零件吗？具体怎么接？
- *混搭* — 论文的某个组件和你已有的东西组合能产生新东西吗？产出什么？
- *反转* — 论文的做法和你的默认假设相反吗？该停下什么、开始什么？

每个命中的视角都给一个能动手的小动作（"明天可以试试 X"），不要停在"值得思考"。一两条扎实的启发，比五条"也许可以"有用得多。

#### 九、关键引用（双语对照）

挑论文里 3-5 处*真正重要的句子*——通常是核心 claim、方法 insight、关键定义、limitation 声明。每处保留*原文 + 中文翻译*：

```
[Section X.Y] "Original verbatim quote here..."

译文：xxx（用大白话翻译，不要直译）
```

挑选标准：抽出来贴在研究笔记本上能记住一年的句子才算。论文摘要的客气话不算；方法描述的程序性段落不算；只有*真正定义了思想形状*的句子才挑出来。

HTML 里这部分用左侧色条 + 浅底色突出，原文与译文上下排列。

#### 十、Q&A

模拟一个对这个方向有研究的面试官，审视读者是否真懂这篇论文。*5-10 道就够*——挑论文里最值得深挖的几个点出题，不必凑数，宁可少而精。

问题主要围绕以下几类（哪类适用就出哪类，不必每类都覆盖）：

- *模型结构* — 模型整体长什么样？由哪些模块组成？模块之间怎么连？
- *输入输出* — 模型的输入是什么形态（文本 / 图像 / token 序列 / 向量 / 多模态…）？输出又是什么？维度、形态、含义分别是？
- *训练数据* — 用了什么数据？数据从哪来？怎么收集 / 构造 / 清洗 / 标注的？规模多大？数据本身有什么特别之处？
- *训练流程* — 训练分几个阶段（预训练 / SFT / RLHF / DPO / 蒸馏 / 课程学习…）？每个阶段做什么？为什么要分阶段？阶段间怎么衔接？
- *训练策略* — 用了哪些关键策略？（loss 设计、采样策略、数据配比、学习率调度、正则手段、奖励建模…）每个策略解决什么问题？
- *具体方法* — 论文的核心方法是什么？*怎么理解*这个方法在做的事？（用类比、用直觉、用变形——别只是复述步骤）。为什么这么设计而不是别的方式？
- *实验细节* — 在什么数据上评测？baseline 选了哪些、为什么是这些？关键超参怎么设的？硬件 / 算力规模？训练多长时间？复现需要注意什么坑？
- *实验结果引发的思考* — 某个数字反映了什么本质问题？某个对照实验意外地证明 / 推翻了什么？哪个结果让人意外、意味着什么？这个发现对领域的隐含影响是什么？

也可以包含一些开放性的延伸问题：方法什么时候会失效？哪个假设最薄弱？能迁移到 XX 场景吗？和 YY 方法结合会怎样？读完这篇，你对 ZZ 问题的看法变了吗？

每个问题都给*参考答案*——不是标答，是引导思路。答案要扎实、能站住脚，不能含糊敷衍。开放性问题给出*作者自己的思考方向*而非"见仁见智"。

HTML 里 Q&A 推荐做成可折叠（accordion）：点击问题展开答案，鼓励读者先想再看。

格式：

```
*Q1*：xxx？
*A*：xxx
```

### 3. 过红线

逐条扫红线（每条都过）。额外检查：

- 破公式——否定式排比全文不超过两处，三段式改两项或四项
- 变节奏——长短句交替
- 杀金句——听起来像可引用的，重写
- 查跳跃——逻辑每步可追
- 公式都翻译——每条 LaTeX 后面有中文一句话说明

列修改清单确认后再生成 HTML。

### 4. 生成 HTML 阅读版

调用 `frontend-design` 技能生成单文件 HTML。把以下三类材料一并打包传给它：

1. *内容*：摘要 + 2.1-2.10 的全部正文（包括 LaTeX 公式、SVG 代码、引用块、Q&A 列表）
2. *资源路径*：从论文提取的配图绝对路径或 base64 数据
3. *视觉风格规范*（见下）

*保存路径*（按 `## 格式约束 → 文件命名` 规则）：
- 有本地 PDF → 与 PDF 同级，文件名 `{简短标题}.html`
- 下载了 arxiv PDF → `~/Documents/notes/pdf/{identifier}--paper-{简短标题}.html`
- 纯网页来源 → `~/Documents/notes/html/{identifier}--paper-{简短标题}.html`

*HTML 结构*：

```
<header>
  <h1>{思想骨句}</h1>
  <p class="subtitle">{论文原标题 / 必要时中文兜底} — {作者}, {venue}</p>
  <p class="meta">{日期} · <a href="{源链接}">原文</a> · <a href="{PDF 路径}">PDF</a></p>
</header>

<main>
  <section id="abstract">摘要（200-300 字全景导览，进入正文前的地图）</section>
  <section id="motivation">一、Motivation</section>
  <section id="problems">二、现存问题</section>
  <section id="related">三、相关工作翻译</section>
  <section id="method">四、方法详解（含配图、SVG、公式）</section>
  <section id="experiments">五、实验结果（含 SVG 对比图）</section>
  <section id="summary">六、总结</section>
  <section id="insight">七、Insight（按需出现）</section>
  <section id="inspiration">八、启发（按需出现）</section>
  <section id="quotes">九、关键引用（双语对照）</section>
  <section id="qa">十、Q&A（accordion）</section>
</main>

<aside id="toc">侧边目录（桌面端固定，移动端折叠）</aside>
```

*视觉风格要求*（必须传给 frontend-design）：

- *学术阅读风* — 干净、留白充足、行距 1.7+、正文 16-18px
- *中文衬线字体*（Noto Serif CJK / 思源宋体），代码 / 公式用等宽
- *KaTeX 渲染 LaTeX*（CDN 引入 katex.min.js + auto-render.min.js）
- *配色简洁* — 背景偏白或暖白（#fafaf7）、正文深灰（#2c2c2c）、强调色一种（如 #4a6fa5）、辅助灰
- *双语引用块* — 左侧 4px 色条 + 浅底色（#f5f5f0）；原文上、中文译文下，中间一条细分隔线
- *Q&A accordion* — 默认折叠，鼠标悬停问题有微妙背景变色，点击平滑展开
- *SVG 内嵌* — 不依赖外部 chart 库，用 `viewBox` 实现响应式
- *配图自适应* — `max-width: 100%; height: auto`
- *侧边目录*（桌面端 ≥1024px）— 滚动时高亮当前 section
- *移动端响应式* — 侧边目录折叠到顶部下拉
- *单文件* — 所有 CSS 内联在 `<style>`，外部依赖只通过 CDN（KaTeX、字体），保证 HTML 可独立打开

调用方式：

```
Skill: frontend-design
任务：生成单文件 HTML 论文阅读笔记
内容：[2.1-2.10 完整正文 + SVG 代码 + 配图路径]
风格：[上面的视觉风格要求]
输出路径：[按规则确定的绝对路径]
```

生成完毕后报告 HTML 绝对路径与文件大小。

## 验收

- *h1 朴素* — `{论文简短名} 阅读笔记` 即可，不追求凝练；副标题给完整原标题 + 作者 + 年份
- *摘要先行* — 正文最前面有一段 200-300 字的整体介绍，覆盖"做了什么 / 解决什么 / 核心方法 / 关键结果"四要素，不剧透洞见，外行扫一遍能形成三句话级别的认知
- *锚点贯穿* — Motivation 立的具象场景在方法详解、洞见里被反复回到，没有换地图
- *Motivation 落地* — 用具体场景而非"为提升性能"等空话；前作短板讲到机制层
- *方法详解配图齐* — 关键架构 / 流程图已提取并嵌入；*每张结构图边界完整、清晰可读、单张 ≤ 300 KB*（无截边、无遮挡、文字看得清、HTML 不臃肿）；公式后必有中文翻译跟随
- *SVG 数据准确* — 每个数字都能在论文 Table/Figure 里找到对应单元格；末尾标 "Source: ..."；不四舍五入、不混淆 open-source / closed-source 对比段
- *实验有 SVG* — 至少一张承重 SVG（消融对比 / 主结果 / 流程图），不是把表格搬过来
- *Insight 真才写* — 七节如果写了，那句话单独抽出来要立得住；论文没思想火花就略掉或一句话说明，不硬挤
- *启发要能动手* — 八节如果写了，落点是"你可以___"而非"值得思考___"；连接不上就略掉，宁缺毋滥
- *Q&A 5-10 道、围绕实质问题* — 覆盖模型结构 / 输入输出 / 训练数据 / 训练阶段策略 / 具体方法 / 实验细节 / 实验结果思考等关键面向（哪类适用出哪类，不必每类都覆盖），每题有扎实参考答案
- *双语关键引用* — 3-5 处关键句保留原文 + 中文译文（不是直译）
- *外行能跟* — 不懂这个领域的聪明人读完能复述三件事：问题（具体到一个例子）/ 解法（机制+设计理由）/ 核心发现（含反直觉副发现）；如果有 Insight / 启发，再加上"能带走什么"
- *公式都翻译* — 每条 LaTeX 公式后有一句中文白话说明
- *公式都渲染* — *打开 HTML 后肉眼逐条确认所有公式都正常渲染*，没有 `$...$` 裸露、没有红色报错、符号完整无缺失
- *HTML 可读* — 单文件可独立打开（双击即用），配图与 SVG 显示，Q&A 折叠展开，移动端不破版
- *HTML 不臃肿* — 单文件总大小最好 ≤ 2 MB，超过就压缩配图
- *零割裂感* — 读完像一个人在跟你说「我读了篇论文，发现了个有意思的事」
